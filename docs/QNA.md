# 질의응답 대비 문서 (Q&A)

발표 또는 코드 리뷰 시 예상되는 질문과 답변 모음입니다.

---

## Thread / 동시성

**Q. 왜 raw Thread 대신 javax.swing.Timer를 사용했나요?**

> Swing은 모든 UI 업데이트를 **EDT(Event Dispatch Thread)** 라는 단 하나의 스레드에서 처리해야 합니다. raw Thread에서 직접 `lblTimeDisplay.setText()`를 호출하면 EDT 위반으로 화면이 깨지거나 프로그램이 멈출 수 있습니다.
> `javax.swing.Timer`는 내부적으로 스레드를 사용하지만, 콜백(`tick()`)을 EDT 위에서 자동으로 실행시켜줍니다. 결과적으로 Thread를 활용하면서도 안전한 구조입니다.
> raw Thread를 쓰려면 매 업데이트마다 `SwingUtilities.invokeLater()`로 EDT에 위임해야 하므로 코드가 복잡해집니다.

---

**Q. Thread.sleep()은 어디서 사용되나요?**

> 직접 `Thread.sleep()`을 호출하지 않습니다. `javax.swing.Timer`가 내부적으로 동일한 역할을 수행합니다. `new Timer(1000, e -> tick())`은 "1초마다 tick()을 EDT에서 실행"을 의미하며, 이것이 `Thread.sleep(1000)` + `SwingUtilities.invokeLater()`를 대체합니다.

---

**Q. 타이머가 작동 중일 때 UI가 멈추지 않는 이유는?**

> `Timer`는 별도의 타이머 스레드에서 동작하고, 콜백만 EDT로 전달합니다. EDT는 타이머 콜백을 처리하는 짧은 시간 외에는 계속 사용자 입력을 받을 수 있어 UI가 멈추지 않습니다.

---

## Collection (컬렉션)

**Q. ArrayList를 선택한 이유는?**

> 할 일 목록은 사용자가 실시간으로 추가/삭제하므로 크기가 고정된 배열(`Todo[]`)을 쓰면 미리 크기를 잡거나 복사-재할당을 직접 구현해야 합니다. `ArrayList`는 동적으로 크기가 늘어나므로 구현이 단순해지고, 추가(`add`)는 O(1) amortized로 성능도 충분합니다.

---

**Q. HashMap은 어디서 사용되나요?**

> `DataManager.loadAllStats()`에서 날짜(`"2026-05-24"`)를 Key, `PomodoroStats` 객체를 Value로 저장합니다. 날짜로 특정 날의 통계를 O(1)로 조회할 수 있어 캘린더 뷰에서 빠르게 기록 유무를 판단합니다.

---

**Q. Collections.unmodifiableList()는 왜 사용하나요?**

> `TodoService.getTodos()`는 내부 `todos` 리스트의 참조를 반환합니다. 외부에서 이 리스트에 직접 `add()`나 `remove()`를 호출하면 `saveTodos()`가 호출되지 않아 파일과 메모리 상태가 불일치합니다. `unmodifiableList()`로 감싸면 수정 시 예외가 발생해 실수를 방지합니다.

---

## File I/O

**Q. 왜 데이터베이스(DB) 대신 텍스트 파일을 사용했나요?**

> 이 프로젝트는 단일 사용자가 로컬에서 실행하는 데스크톱 앱입니다. DB를 쓰면 MySQL, SQLite 등 외부 드라이버를 설치하고 연결 설정을 해야 하는 반면, 텍스트 파일은 JDK만 있으면 추가 설치 없이 동작합니다. 데이터 규모(하루 수십 건)도 파일로 충분히 처리 가능합니다.

---

**Q. 파일 형식으로 왜 CSV가 아닌 pipe(`|`)로 구분했나요?**

> CSV(쉼표 구분)를 쓰면 할 일 제목에 쉼표가 포함될 때 파싱이 깨집니다. Pipe(`|`)는 일반 사용자가 입력할 가능성이 낮아 더 안전합니다. 실제로 `split("\\|", -1)` 또는 `split("\\|", 2)`로 파싱합니다.

---

**Q. 파일을 매번 전체 재저장하는 방식이 비효율적이지 않나요?**

> 현재 데이터 규모(할 일 수십 건, 날짜별 통계)에서는 파일 크기가 수 KB 이하로 성능 문제가 없습니다. append 방식은 삭제/수정 처리가 복잡해지므로, 단순함과 안정성을 위해 전체 재저장 방식을 선택했습니다.

---

**Q. 앱이 비정상 종료되면 데이터가 손실되지 않나요?**

> `saveTodos()`와 `saveStats()`는 데이터가 변경될 때마다 즉시 호출됩니다(추가/삭제/완료/타이머 완료 시). 메모리와 파일이 항상 동기화되어 있으므로 언제 종료해도 마지막 변경까지 보존됩니다.

---

## 예외 처리

**Q. try-with-resources를 사용한 이유는?**

> `try (BufferedReader reader = ...)` 구문은 블록이 정상 종료되거나 예외가 발생해도 자동으로 `reader.close()`를 호출합니다. 직접 `finally`에서 닫는 방식보다 코드가 짧고 실수가 없습니다.

---

**Q. 파일이 없을 때 어떻게 처리하나요?**

> 앱 최초 실행 시 `data/` 폴더와 파일이 없을 수 있습니다. 각 `load` 메서드는 `Files.exists()`로 파일 존재 여부를 먼저 확인하고, 없으면 빈 컬렉션이나 기본값을 반환합니다. 파일 읽기 중 예외가 발생해도 `catch (IOException e)`에서 안내 메시지를 출력하고 기본값을 반환하므로 앱이 강제 종료되지 않습니다.

## 설계 결정

**Q. CardLayout을 사용하지 않은 이유는?**

> PRD에서 CardLayout을 제안했지만, 타이머·투두리스트·캘린더를 동시에 보는 것이 사용성 측면에서 더 직관적이라고 판단해 3패널 동시 표시 방식으로 변경했습니다. 화면이 충분히 넓고(1280px) 각 기능이 독립적이므로 전환 없이 한 화면에서 모두 확인할 수 있습니다.

---

**Q. 왜 StatsPanel에 JTable을 사용했나요?**

> 집중 시간·완료·취소·성공률처럼 여러 지표를 정렬된 형태로 보여줄 때 JTable이 가장 적합합니다. `DefaultTableModel`을 사용해 `refreshTable()`로 데이터를 동적으로 갱신할 수 있고, `isCellEditable()`을 오버라이드해 사용자가 직접 수정하는 것을 막았습니다.

---

**Q. 포모도로 성공률은 어떻게 계산하나요?**

> `성공률(%) = 완료 횟수 / (완료 횟수 + 취소 횟수) × 100`
> 완료/취소가 모두 0이면 0.0%를 반환해 `ArithmeticException`을 방지합니다. 취소는 타이머가 진행 중일 때 RESET 버튼을 누를 때만 카운트되며, 0:00 자동 완료 시에는 취소로 집계되지 않습니다.

---

**Q. 레이어(계층)를 나눈 이유는?**

> UI(`ui.*`)가 파일을 직접 읽으면, 파일 형식이 바뀔 때 UI 코드도 수정해야 합니다. Service(`service.*`)가 비즈니스 로직을 담고, Storage(`storage.*`)가 파일 입출력을 전담하면 각 레이어가 독립적으로 수정 가능합니다. 예: 나중에 txt 대신 JSON으로 저장 방식을 바꿔도 `DataManager`만 수정하면 됩니다.

---

## 캘린더

**Q. 달력의 시작 요일(일요일)은 어떻게 계산하나요?**

> `YearMonth.now().atDay(1).getDayOfWeek().getValue()`는 월=1, 일=7을 반환합니다. 여기에 `% 7`을 하면 일=0이 되어 일요일 시작 달력에 맞는 빈 칸 개수를 구할 수 있습니다.

---

**Q. 기록이 있는 날짜는 어떻게 표시하나요?**

> `DataManager.loadAllStats()`가 반환한 `Map<String, PomodoroStats>`에서 해당 날짜 키가 존재하면 버튼 배경을 연분홍(`#FEE2E2`)으로, 오늘은 빨간(`#B42029`)으로, 선택된 날짜는 파란색(`#3B82F6`)으로 구분합니다.
